# Conscio v0.5 "Cognitive Modes" — Design Spec

**Date:** 2026-06-06
**Version target:** 0.4.0 → 0.5.0
**Theme:** The agent knows *which mode it's in*, *where it's heading*, and *which memories matter most*.

Completes the four roadmap modules left after v0.4's Self-Judgment trio:
**#1 Shard Engine**, **#5 Trajectory Vector**, **#4 Content Layering**.
(**#6 Coherence Check** deferred to v0.6 — its semantic-Jaccard dedup overlaps the
shipped literal `DedupBlocks` stage and risks false-merging distinct heartbeat blocks.)

**Constraints carried from v0.4:** zero schema migration, zero new dependencies,
WorldModel stays pure-JSON, ContentStore FTS5 tables untouched. Every new behavior
is additive and advisory — none feeds drives or goals.

## Module A — Shard Engine (#1)

**Origin:** Cognitive Shards (Noosphere-Manifold). Seven cognitive modes; the agent
infers which one it is currently operating in from recent activity.

**New file:** `conscio/shard_engine.py`

### A1. The enum

```python
class Shard(Enum):
    ARCHITECT       = "ARCHITECT"
    ENGINEER        = "ENGINEER"
    JANITOR         = "JANITOR"
    SECURITY_ANALYST = "SECURITY_ANALYST"
    ARCHAEOLOGIST   = "ARCHAEOLOGIST"
    EXPERT_CODER    = "EXPERT_CODER"
    DREAMER         = "DREAMER"
```

### A2. Keyword table (disjoint, whole-word, `len >= 3`, lowercase)

Sets are **disjoint** — no keyword appears in two shards — so a single event word
votes for exactly one shard (deterministic tallies).

| Shard | Keywords |
|-------|----------|
| `ARCHITECT` | design, architecture, plan, blueprint, schema |
| `ENGINEER` | implement, build, feature, code, develop |
| `JANITOR` | refactor, cleanup, prune, tidy, dedupe, lint |
| `SECURITY_ANALYST` | bug, vulnerability, security, exploit, error, anomaly, cve |
| `ARCHAEOLOGIST` | research, investigate, explore, history, origin |
| `EXPERT_CODER` | debug, trace, diagnose, fix, stacktrace |
| `DREAMER` | dream, crystallize, consolidate, friction, release |

`DREAMER` fires on DreamCycle-emitted vocabulary; it is the rarest shard by design
(only triggers when a dream cycle has run recently).

### A3. Event → scannable text  *(resolves Ressalva #1)*

`Event.data` is a **dict**, not text. Scanning `json.dumps(data)` would let structural
keys (`confidence`, `priority`, `meta_confidence`) generate spurious hits. So we scan
**only the values**, recursively, plus the event `type`:

```python
def _event_text(event: dict) -> str:
    """type + recursively-flattened data VALUES (keys excluded)."""
    parts: list[str] = [str(event.get("type", ""))]

    def _flatten(v) -> None:
        if isinstance(v, dict):
            for sub in v.values():      # values only — keys are structural metadata
                _flatten(sub)
        elif isinstance(v, (list, tuple)):
            for sub in v:
                _flatten(sub)
        else:
            parts.append(str(v))

    _flatten(event.get("data", {}))
    return " ".join(parts).lower()
```

Example: `{"type": "error", "data": {"confidence": 0.8, "anomalies": ["timeout bug"]}}`
→ text `"error 0.8 timeout bug"` → votes `SECURITY_ANALYST` (matches `error`, `bug`).
The key `confidence` and value `0.8` produce no shard hit.

### A4. Inference — pure function

```python
def infer_shard(events: list[dict], window: int = 20) -> Optional[Shard]:
    """
    Tally whole-word keyword hits across the last `window` events.
    Returns the dominant Shard, or None when no keyword matches at all.

    - events: newest-last list of Event.to_dict() payloads.
    - Whole-word match: r"\b{re.escape(kw)}\b" (same hardening as DreamCycle friction).
    - Tie-break: the shard whose keyword appeared in the MOST RECENT event wins.
    - No matches anywhere -> None (advisory; heartbeat line omitted).
    """
```

Determinism guarantees:
- Same event list → same shard, always (no clock, no randomness).
- Disjoint keyword sets → no double-counting.
- Tie-break by recency is fully ordered → never ambiguous.

### A5. Stateful wrapper + transition event

```python
class ShardEngine:
    def __init__(self, event_bus, window: int = 20):
        self._bus = event_bus
        self._window = window
        self.current: Optional[Shard] = None

    def update(self, events: list[dict]) -> Optional[Shard]:
        """Re-infer; emit shard:transition on the bus only when it CHANGES."""
        new = infer_shard(events, self._window)
        if new is not None and new != self.current:
            self._bus.emit(
                type="system", category="consciousness",
                data={"shard_transition": True,
                      "from": self.current.value if self.current else None,
                      "to": new.value},
                priority=PRIORITY_NORMAL,
            )
            self.current = new
        return self.current
```

`infer_shard` (pure) is tested independently from `update` (side-effecting). First call
with a real shard transitions from `None`; `None` inferences never clear a set shard
(advisory stability — a quiet window keeps the last known mode).

### A6. Heartbeat injection (live state — `context_manager.py`)

Mirrors the existing `metabolic` / `reflection_quality` advisory fields:
- `ConsciousnessState.shard: str = ""` field.
- `to_injection()`: `▷ shard: ENGINEER` — emitted only when non-empty **and**
  `context_mode != MINIMAL`.
- `build_state(..., shard: str = "")` param + passthrough.

The engine computes the active shard from `event_bus` recent events and passes the
shard's `.value` (or `""` when `None`) to `build_state`. **Advisory only** — the shard
never feeds drives, goals, or reflection.

## Module B — Trajectory Vector (#5)  *(+ latent bug fix)*

**Origin:** Soul Package / Temporal Bridge (Noosphere). Carries *direction* and
*texture* across the session boundary.

### B1. Soft fields on `SessionSummary` (`session_lifecycle.py`)

```python
trajectory: str = ""        # code-owned — where the agent is heading
vibes: str = ""             # LLM-only — emotional texture ("frustrated but progressing")
identity_anchor: str = ""   # LLM-only — processing style ("methodical debugger")
```

### B2. Source-of-truth split  *(resolves Ressalva #2)*

Two potential writers (the LLM heartbeat author vs. `enrich_with_conscio` code).
Resolved by ownership, not by empty-checks:

- **`trajectory` is code-owned.** `enrich_with_conscio` **always overwrites** it —
  the code-derived direction is more current than the last heartbeat's text.
  Objective and derivable (goals + shard), so code is authoritative.
- **`vibes` and `identity_anchor` are LLM-only.** Code **never** touches them.
  Subjective texture only the heartbeat author can judge; they survive enrichment
  untouched and backfill empty for old summaries.

### B3. `enrich_with_conscio` derivation

```python
# trajectory: mode + direction. Always overwrite.
shard = _active_shard_value(engine)          # "" when None
top_goal = summary.active_goals[0] if summary.active_goals else ""
if shard or top_goal:
    arrow = f"{shard} → {top_goal}".strip(" →")
    summary.trajectory = arrow               # e.g. "ENGINEER → ship recall layering"
```

### B4. Latent bug fix — `WorldModel.list_entities`

`session_lifecycle.py:235` calls `engine.world.list_entities(limit=5)`, which **does
not exist** on `WorldModel` — the line is silently swallowed by its `except Exception:`.
This module fixes it (the call sits in the `enrich` path being modified anyway):

```python
def list_entities(self, limit: int = 5) -> list[dict]:
    """Top-N entities by relevance (desc). Each dict carries its 'name'."""
    items = [
        {"name": name, **info}
        for name, info in self._data["entities"].items()
    ]
    items.sort(key=lambda e: e.get("relevance", 0.0), reverse=True)
    return items[:limit]
```

This satisfies the existing consumer: `f"{e['name']}:{e.get('state', '?')}"`.

### B5. Heartbeat / handoff formatting  *(resolves Minor #2 — budget)*

`format_heartbeat` is bounded by `HB_MAX_CHARS = 1400` with a hard-truncate backstop.
New fields are placed **early** (right after `Título`, before the long intent/action
lists) so they survive truncation, each capped at `[:80]`:

```python
# in format_heartbeat, immediately after the Título line block:
if summary.trajectory:
    lines.append(f"**Trajetória:** {summary.trajectory[:80]}")
if summary.vibes:
    lines.append(f"**Vibe:** {summary.vibes[:80]}")
if summary.identity_anchor:
    lines.append(f"**Âncora:** {summary.identity_anchor[:80]}")
```

`format_handoff` (the richer doc, no tight char cap) includes the same three fields,
uncapped, inside the existing `## 🧬 Estado Conscio` section. The 1400-char backstop
at the tail of `format_heartbeat` remains the final guard.

## Module C — Content Layering (#4)  *(derive-at-query, no migration)*

**Origin:** Noetic Helix layers (Noosphere) — ROUTINE (N-1), PROCESSING (N),
INTUITION (N+1). ContentStore uses **FTS5 virtual tables**, which cannot be `ALTER`ed,
so the layer is **derived at query time** from fields already returned per result.
Zero schema change, zero reindex.

**New file:** `conscio/content_layer.py`

### C1. The enum + pure classifier

```python
class ContentLayer(Enum):
    ROUTINE    = "routine"      # N-1: factual/system noise
    PROCESSING = "processing"   # N:   insights, reflections, decisions
    INTUITION  = "intuition"    # N+1: hypotheses, predictions, anomalies

_LAYER_BY_CATEGORY = {
    "system":   ContentLayer.ROUTINE,
    "trading":  ContentLayer.ROUTINE,
}
_LAYER_BY_CONTENT_TYPE = {
    "prediction": ContentLayer.INTUITION,
    "anomaly":    ContentLayer.INTUITION,
    "reflection": ContentLayer.PROCESSING,
    "insight":    ContentLayer.PROCESSING,
}

def layer_of(category: str, content_type: str) -> ContentLayer:
    """content_type wins over category (more specific). Default PROCESSING."""
    if content_type in _LAYER_BY_CONTENT_TYPE:
        return _LAYER_BY_CONTENT_TYPE[content_type]
    if category in _LAYER_BY_CATEGORY:
        return _LAYER_BY_CATEGORY[category]
    return ContentLayer.PROCESSING
```

`content_type` is checked first because it is the more specific signal (a `prediction`
in the `consciousness` category is INTUITION, not PROCESSING).

### C2. Layer priority + non-destructive reorder in `recall()`  *(Minor #1)*

`SearchResult` already carries `source_category`, `content_type`, and `rank` (RRF
score, **higher = better**). The reorder lives in **`engine.recall()` only** —
`ContentStore.search()` keeps its contract, so existing search tests do not regress.

Layer is a **near-tie tiebreak**, never a primary sort — a highly-relevant ROUTINE
hit is never buried under a barely-relevant PROCESSING hit.

```python
LAYER_EPSILON = 0.01            # rank-bucket width (~1% of RRF range)
_LAYER_PRIORITY = {
    ContentLayer.PROCESSING: 2,
    ContentLayer.INTUITION:  1,
    ContentLayer.ROUTINE:    0,
}

def _layer_sort_key(r):       # bucket rank, then prefer higher layer within bucket
    layer = layer_of(r.source_category, r.content_type)
    bucket = int(r.rank / LAYER_EPSILON)     # higher rank -> higher bucket
    return (-bucket, -_LAYER_PRIORITY[layer], -r.rank)
```

`recall()` collects `SearchResult`s, sorts by `_layer_sort_key` (stable), then emits
snippets. Behavior:
- rank 0.030 ROUTINE vs 0.029 PROCESSING (ε=0.01) → same bucket → **PROCESSING first**.
- rank 0.030 ROUTINE vs 0.010 PROCESSING → different buckets → **ROUTINE first** (relevance wins).

**Bucket-0 collapse (intentional).** `int(r.rank / LAYER_EPSILON)` sends every low rank
into bucket 0: `int(0.009 / 0.01) == 0`, same as `rank == 0.0`. So all near-zero scores
share the worst bucket and are ordered purely by layer priority — acceptable because low
FTS ranks are noise anyway. A future tighter ε is a one-line change. Boundary covered by
the `rank == 0.0` test below.

The SessionRAG semantic branch is appended after the ContentStore branch as today
(its results have no FTS rank to bucket); layering applies to the ContentStore path.

## Order & Dependencies

```
Shard Engine ─────────────► enrich_with_conscio (trajectory derives from shard)
Trajectory Vector ────────► SessionSummary + heartbeat/handoff
Content Layering ─────────► recall() reorder   (independent — ships last)
```

Build order: **Shard → Trajectory → Layering.** Each task ships self-contained and
green; Trajectory depends on Shard for the trajectory derivation.

## Testing (RAM-safe — never the full suite)

Per the standing constraint (the dev box OOMs on `pytest tests/`), every task is
verified with **its own file, one process at a time**:

- `tests/test_shard_engine.py` — `infer_shard` determinism per event pattern; whole-word
  (no substring false-positive); values-only scan (key `confidence` ≠ hit); tie-break by
  recency; no-match → `None`; `update` emits `shard:transition` only on change.
- `tests/test_trajectory_vector.py` — fields present + serialized; LLM-only fields survive
  enrich; trajectory overwritten by enrich; `list_entities` returns top-N by relevance;
  heartbeat places fields early + within `HB_MAX_CHARS`; old-summary backfill graceful.
- `tests/test_content_layer.py` — `layer_of` per category/content_type incl. default;
  content_type-wins-over-category; recall near-tie reorder by layer; highly-relevant
  ROUTINE not buried (boundary at ε); **rank 0.0 ROUTINE vs 0.0 PROCESSING → PROCESSING
  first** (bucket-0 collapse, layer priority decides); unknown category → PROCESSING.

Touched-module regressions run sequentially when needed:
`test_context_manager.py`, `test_session_lifecycle.py`, `test_engine_recall.py`.
